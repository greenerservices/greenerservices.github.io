6. Develop efficient software

Greener software generally means efficient applications that use less energy on servers, supporting infrastructure and end-user devices.

Further Reading

10 Recommendations for Green Software Development, Green Software Foundation

Key Disciplines

Development

6a. Create efficient code

Adopt a range of strategies to reduce software’s processing requirements, including choosing programming languages that are energy efficient and keep code clean and simple.

Lifecycle phases

Alpha Beta Live

Actions

There are a range of actions that could potentially be taken to support this guideline. For more detailed guidance see the box below.

(i) Choose greener coding languages where possible

Different programming languages run at different speeds, with significant implications for energy use. Byte for byte, JavaScript (JS) has a much bigger impact than HTML, CSS, images or text, because it causes energy-intensive processing to occur on the user’s device.

NB: There may be a tradeoff between the ‘greenness’ of code and team productivity

We need to factor in levels of skill/familiarity of programmers with particular languages, and so dev time, effort and maintenance into decision making about which coding languages are used.

'You might get a hundredfold reduction in carbon emissions by switching, for example, from Python to a much more efficient language like Rust, but there will be a price to pay in productivity. Developers really do deliver much more quickly when they are using lower machine-efficiency languages like Python. As a result, writing efficient code is unattractive to businesses, who want to devote their developer time to building new features, not writing more streamlined code.'

Currie, Anne; Hsu, Sarah; Bergman, Sara. Building Green Software (pp. 24-25). O'Reilly Media. Kindle Edition.

(ii) Only cache when necessary

Develop a caching strategy for the specific application context and usage scenario as that can lead to significant energy efficiencies. Make use of public and shared caches, for example with the GOV.UK assets.

(iii) Only load resources and components when they are actually used

Loading unused components means needless data transfer to user device.

(iv) Choose static over dynamic pages where possible

A flat HTML file is fast to serve and uses considerably less energy compared to webpages that are generated dynamically. This is especially important for services with high volumes of users.

More detailed guidance on greener software development

There are a variety of publicly available guidelines that describe patterns and techniques that can be used to help implement these principles.

Green Software Foundation Patterns

The Green Software Foundation have developed a well-researched and regularly updated set of patterns and techniques to reduce the environmental impacts of digital services.

W3C Web Sustainability Guidelines

The W3C's draft Web Sustainability Guidelines, with comprehensive recommendations for making websites and products more sustainable. Includes guidance under the following categories: user experience design, web development, hosting and infrastructure and business and product management considerations).

Handbook of Sustainable Design of Digital Services

The Handbook of Sustainable Design of Digital Services, known as GR491, created in France by Institutes for Sustainable IT, includes a similar range of recommendations for teams to consider.

Measurement

The Software Carbon Intensity (SCI) defines a standardised methodology for calculating the rate of carbon emissions for a software system. It generates a score, designed to ‘help software practitioners make better, evidence-based decisions during system design, development, and deployment, that will ultimately minimize carbon emissions.’

The SCI encompasses energy consumption, resource utilization, and emissions generated throughout the software lifecycle and has now become an International Organisation for Standardisation (ISO) standard.

6b. Re-use lean software

Actions

(i) Reuse common components

Common components can be used on both front and back end as well. Re-use can include modularisation, libraries, or managed services.

Code re-use can lead to quicker delivery time, reduced costs and reduced environmental impacts.

(ii) Remove unused features

Remove unused software features and unnecessary logging, looping and polling. Unused software features still consume energy.

(iii) Conduct regular code reviews

Conduct regular code reviews to assess progress and features.
